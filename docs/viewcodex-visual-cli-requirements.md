# Viewcodex 可视化 Codex CLI 需求文档

## 1. 产品意图

Viewcodex 是一个轻量级的 Codex CLI 可视化工作台。

它不替代 Codex CLI，而是让日常使用 Codex CLI 更舒服，重点解决两个问题：

- 切换项目麻烦。
- 不清楚当前 Codex 线程或进程有没有关闭、是否还在运行。

这个产品应该简单、快速、实用。第一版不追求做成 IDE、插件市场、云产品或复杂管理平台。

## 2. 目标用户

主要用户：

- 经常在本地使用 Codex CLI 的开发者。
- 希望更方便地选择项目、启动 Codex、查看活跃会话，并快速回到 CLI 线程。
- 更想要一个小工具，而不是完整 IDE。

次要用户：

- 其他希望更方便使用 Codex CLI 的用户。

## 3. 核心使用流程

第一版理想流程：

1. 打开 Viewcodex。
2. 手动选择一个项目目录。
3. 选择当前已有的 skill。
4. 选择是否每次任务完成后自动提交 Git。
5. 选择模型。
6. 查看当前是否有正在运行的 Codex 进程。
7. 查看当前上下文使用量，单位显示为 K tokens。
8. 输入自己的需求。
9. 点击确认后运行 Codex。
10. 点击具体线程或会话，进入真实 CLI。
11. 开启某个项目的新会话前，先读取用户配置的规范文档和项目说明文档。

## 4. 第一版范围

### 4.1 项目选择

Viewcodex 应该支持：

- 添加本地项目目录。
- 从界面选择本地项目目录。
- 运行 Codex 前切换和确认当前项目。
- 记住最近使用的项目。
- 从选中的项目目录启动 Codex。

项目选择必须是第一版核心入口。用户不应该手动改配置文件来切换项目。

### 4.2 Skill 选择

Viewcodex 应该支持：

- 展示当前本地可用的 skill。
- 运行 Codex 前选择 skill。
- 将选中的 skill 纳入启动命令或提示词流程。

第一版只需要使用本地已有 skill，不需要下载、安装、排序或管理 skill 市场。

### 4.3 Codex 进程可见性

Viewcodex 应该让用户清楚看到当前是否有 Codex 正在运行。

至少应该展示：

- 活跃 Codex 进程。
- 能检测到时，显示对应项目目录。
- 基础状态，例如运行中、已退出、未知。
- 能进入对应 CLI 会话的入口。

第一版至少要完整追踪 Viewcodex 自己启动的 Codex 会话；外部终端手动启动的 Codex 进程可以后续再接入。

### 4.4 线程 / CLI 入口

Viewcodex 应该允许用户点击可见的线程或会话，并进入对应 CLI。

CLI 仍然是真正的交互界面。Viewcodex 是启动器和导航器，不是聊天界面的替代品。

### 4.5 模型选择

Viewcodex 应该允许用户在启动 Codex 前选择模型。

第一版可以使用简单下拉框或本地配置列表。

### 4.6 上下文使用量显示

Viewcodex 应该在可获得信息时显示当前上下文使用量，单位为 K tokens。

如果无法从 Codex CLI 获得精确值，可以显示：

- 未知
- 上次检测值
- 估算值

如果是估算值，界面不应该伪装成精确值。

### 4.7 Git 提交选项

Viewcodex 应该提供一个简单选项，让用户选择每次任务完成后是否自动提交 Git。

具体实现可以在开发时确定，例如：

- 任务开始前检查 `git status`。
- 任务完成后展示本次变更摘要。
- 用户开启该选项时，自动执行一次 Git commit。

第一版只保留为简单开关。

### 4.8 启动前项目文档阅读

Viewcodex 的界面中应该提供“项目文档 / 启动规范”配置区。

用户可以为每个项目配置：

- 必读规范文档，例如 `AGENTS.md`、`VIEWCODEX_BRIEF.md`。
- 项目说明文档，例如 `README.md`、`docs/project.md`。
- 额外上下文文档，例如需求文档、业务规则、接口说明。

启动文档必须可以在界面中选择和创建：

- 选择：从当前项目目录中选择已有 Markdown 文档。
- 创建：在当前项目中创建新的规范或项目说明文档。
- 创建后自动加入该项目的启动文档列表。

当用户选择一个项目并准备开启新的 Codex 会话时，Viewcodex 必须先读取这些文档。

规则：

- 必读文档不存在时，不启动新会话，并提示用户修复。
- 可选文档不存在时，显示提醒，但允许继续。
- 读取到的文档应作为本次 Codex 启动上下文的一部分。
- 用户可以在界面中修改每个项目的文档配置。

## 5. 第一版不做什么

第一版明确不做：

- 完整 IDE。
- 内置代码编辑器。
- 插件或 skill 市场。
- 复杂 MCP 下载中心。
- 云同步。
- 账号系统。
- 替代 Codex CLI 的 Web 聊天界面。
- 多人协作。
- 高级项目索引。
- 复杂数据看板。

## 6. 推荐技术栈

因为当前目标是自己用得舒服、快速迭代，第一版推荐：

- Electron
- React
- TypeScript
- Vite
- xterm.js
- node-pty
- Zustand
- Tailwind CSS
- lucide-react
- 本地 JSON 配置

如果后面真的要产品化，再考虑 Tauri，用来获得更小安装包和更严格的安全边界。

## 7. 本地配置

第一版可以使用本地 JSON 配置文件，不需要数据库。

建议路径：

```text
~/.viewcodex/config.json
```

建议结构：

```json
{
  "projects": [
    {
      "name": "Viewcodex",
      "path": "/Users/benfit/code/Viewcodex"
    }
  ],
  "models": [],
  "defaultModel": null,
  "gitBeforeRun": true,
  "recentSessions": []
}
```

## 8. 验收标准

第一版达到以下条件即可认为成功：

- 用户可以清楚看到当前是否有活跃 Codex 进程。
- 用户可以直接从界面进入某个线程或会话的 CLI。
- 用户可以在运行 Codex 前选择项目。
- 用户可以在运行 Codex 前选择已有 skill。
- 新会话启动前会读取该项目配置的必读规范和项目文档。
- 界面足够简单，日常使用时不会感觉像一个完整 IDE。

## 9. 决策边界

后续实现时，以下决定可以直接做，不需要再确认：

- 第一版使用 Electron。
- 主界面只做一个启动与会话面板。
- 使用本地 JSON 配置，而不是 SQLite。
- MCP 和 skill 只处理本地已有配置，不做下载中心。
- tmux 支持优先做检测和进入已有 session。
- 项目启动文档配置使用本地 JSON 保存。
- 第一版优先做成一个页面，不拆成复杂多页面应用。
- 优先使用简单开关、下拉框、列表，而不是复杂配置页面。

## 10. 产品原则

Viewcodex 应该围绕这个核心循环设计：

```text
快速打开 -> 选择项目 -> 读取项目规范 -> 选择 skill / 模型 -> 查看活跃 Codex -> 运行或回到 CLI
```

不能明显改善这个循环的功能，都应该延后。
